## 2. 今天为什么学这个

Day 1 的目标不是直接接入复杂框架，也不是一开始就做 RAG、LangGraph 或前端页面。

今天的核心目标是先完成一个最小可运行闭环：

```text
用户输入
  ↓
识别工单问题
  ↓
调用业务工具
  ↓
获取工具返回结果
  ↓
生成结构化异常分析
  ↓
输出处置建议
```

这样可以先验证三个问题：

1. 工业异常分析这个场景是否适合做 Agent。
2. 工具调用链路是否能跑通。
3. 后续是否可以在这个基础上继续扩展 RAG、LangGraph 和评估体系。

---

## 3. 今天理解的核心概念

### 3.1 Agent 和普通 ChatBot 的区别

普通 ChatBot 主要依赖模型根据上下文直接回答。

Agent 不只是回答问题，而是可以：

- 判断任务类型
- 调用工具
- 获取外部数据
- 基于工具结果进行分析
- 输出结构化结论

在本项目中，Agent 不是凭空回答工单异常原因，而是先调用工单、库存、设备工具，再基于工具结果生成分析。

---

### 3.2 ReAct 的理解

ReAct 可以理解为：

```text
Thought → Action → Observation → Answer
```

对应到本项目：

| ReAct 概念 | 项目中的对应 |
|---|---|
| Thought | 判断用户想查询工单还是分析异常 |
| Action | 调用工单、库存、设备工具 |
| Observation | 工具返回的业务数据 |
| Answer | 输出异常分析和处置建议 |

---

### 3.3 Tool Calling 的理解

Tool Calling 的关键不是“写几个函数”，而是让 Agent 能够通过工具获取真实业务数据。

当前 Day 1 中的工具包括：

| 工具 | 作用 |
|---|---|
| `get_work_order` | 查询工单状态、物料、设备、异常 |
| `get_inventory` | 查询库存数量、冻结数量、可用数量、同步状态 |
| `get_device_status` | 查询设备状态、报警信息、是否允许继续生产 |

## 8. 今天遇到的问题

### 问题 1：库存低水位判断错误

一开始输出了：

```text
库存剩余量低于 10%
```

但实际数据是：

```text
总库存: 120
可用库存: 90
```

可用库存比例：

```text
90 / 120 = 75%
```

所以不应该判断为库存不足。

### 解决方式

修正库存判断逻辑：

```text
available_rate = available_qty / total_qty
```

规则：

```text
available_rate < 10%     → 库存严重不足
available_rate < 30%     → 库存偏低
available_rate >= 30%    → 库存数量本身不是主要风险点
```

修正后输出：

```text
当前可用库存比例正常，库存数量本身不是主要风险点。
```
